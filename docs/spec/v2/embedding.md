# Quote Finder Embedding Specification

## 1. Purpose

This document defines the embedding architecture for the Quote Finder revamp.

Embeddings are used exclusively for semantic scene search through:

```text
!qs <scene description>
```

The embedding system converts:

* indexed story chunks
* user scene-search queries

into vectors that can be compared in Qdrant.

The initial implementation uses a small local embedding model running on the Netcup VPS through an ONNX-based runtime.

The embedding system must:

* run without a paid external API
* fit within the available VPS memory
* support CPU inference
* produce consistent document and query vectors
* integrate with the ingestion pipeline
* integrate with semantic search
* expose a replaceable provider interface
* store model identity with every indexed version
* prevent incompatible vectors from being mixed
* support future re-embedding and model migration

---

## 2. Scope

This specification covers:

* embedding-provider boundaries
* local runtime
* initial model selection
* document embedding
* query embedding
* batching
* concurrency
* resource limits
* model caching
* model metadata
* Qdrant vector compatibility
* failure handling
* retries
* observability
* model evaluation
* re-embedding
* model migration
* deployment behaviour

This specification does not cover:

* EPUB parsing
* paragraph extraction
* semantic chunk creation
* Qdrant search ranking
* Discord command formatting
* fic refresh scheduling
* deployment of Neon or Qdrant

Those are defined in separate specifications.

## 3. Initial Decision

The initial embedding implementation will use:

```text
Provider:
local FastEmbed-compatible runtime

Runtime:
ONNX Runtime on CPU

Initial model:
BAAI/bge-small-en-v1.5

Vector dimensions:
384

Execution location:
Netcup VPS

Document embedding:
background ingestion process

Query embedding:
Discord bot search process
```

This is the initial baseline, not a permanent architectural dependency.

The model must be replaceable through the embedding-provider interface.

---

## 4. Why Local Embeddings

The Quote Finder workload is suited to local inference.

Characteristics:

* story ingestion happens occasionally
* refresh ingestion happens infrequently
* semantic queries embed one short string at a time
* request volume is low
* only one or a few fics are expected initially
* the Netcup VPS has 4 GB RAM and two CPU cores
* vectors are stored remotely in Qdrant Cloud
* no GPU is required

Local embeddings avoid:

* per-request API cost
* API quotas
* external provider outages
* network latency for query embedding
* vendor lock-in
* sending story text to another embedding service

The main trade-off is slower batch ingestion compared with a hosted GPU service.

That is acceptable because ingestion is asynchronous.

---

## 5. Core Embedding Rule

Document and query vectors for an active fic version must be generated with the same embedding configuration.

Compatibility requires matching:

```text
provider
model
model revision
vector dimensions
normalization version
query/document mode
```

A semantic query must never search vectors generated by an incompatible model.

## 7. Embedding Provider Interface

All embedding logic must be hidden behind an internal provider interface.

Suggested interface:

```python
from typing import Protocol


class EmbeddingProvider(Protocol):
    @property
    def provider_name(self) -> str:
        ...

    @property
    def model_name(self) -> str:
        ...

    @property
    def model_revision(self) -> str | None:
        ...

    @property
    def dimensions(self) -> int:
        ...

    async def embed_documents(
        self,
        texts: list[str],
    ) -> list[list[float]]:
        ...

    async def embed_query(
        self,
        text: str,
    ) -> list[float]:
        ...

    async def health_check(
        self,
    ) -> bool:
        ...
```

The ingestion and search services must depend on this interface rather than directly importing FastEmbed.

## 9. Local Provider Implementation

Suggested implementation structure:

```python
from collections.abc import Iterable
from fastembed import TextEmbedding


class LocalFastEmbedProvider:
    def __init__(
        self,
        *,
        model_name: str,
        cache_dir: str,
        threads: int,
        batch_size: int,
    ) -> None:
        self._provider_name = "local_fastembed"
        self._model_name = model_name
        self._model_revision = None
        self._dimensions = 384
        self._batch_size = batch_size

        self._model = TextEmbedding(
            model_name=model_name,
            cache_dir=cache_dir,
            threads=threads,
        )

    @property
    def provider_name(self) -> str:
        return self._provider_name

    @property
    def model_name(self) -> str:
        return self._model_name

    @property
    def model_revision(self) -> str | None:
        return self._model_revision

    @property
    def dimensions(self) -> int:
        return self._dimensions

    def embed_documents_sync(
        self,
        texts: list[str],
    ) -> list[list[float]]:
        vectors: Iterable = self._model.embed(
            texts,
            batch_size=self._batch_size,
        )
        return [vector.tolist() for vector in vectors]

    def embed_query_sync(
        self,
        text: str,
    ) -> list[float]:
        vector = next(self._model.query_embed(text))
        return vector.tolist()
```

The production implementation may wrap synchronous inference in a bounded thread executor.

## 10. Async Integration

Local ONNX inference is CPU-bound and typically synchronous.

The Discord event loop must not be blocked directly.

Suggested pattern:

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor


class AsyncEmbeddingProvider:
    def __init__(
        self,
        sync_provider: LocalFastEmbedProvider,
        executor: ThreadPoolExecutor,
    ) -> None:
        self._provider = sync_provider
        self._executor = executor

    async def embed_query(
        self,
        text: str,
    ) -> list[float]:
        loop = asyncio.get_running_loop()

        return await loop.run_in_executor(
            self._executor,
            self._provider.embed_query_sync,
            text,
        )
```

The executor must be bounded.

Suggested query executor size:

```text
1 worker
```

The embedding runtime should not be allowed to occupy arbitrary threads.

---

## 11. Process Separation

Document embedding and query embedding should run in separate process contexts.

### Discord bot process

Responsible for:

* loading the model
* embedding one query at a time
* serving `!qs`
* keeping latency predictable

### Ingestion worker process

Responsible for:

* loading the same model
* embedding thousands of chunks
* batching document embeddings
* exiting after ingestion

Recommended architecture:

```text
quote-finder-bot.service
└── query embedding

quote-finder-refresh.service
└── document embedding
```

The ingestion worker must not run large embedding jobs inside the Discord bot event loop.

---

## 12. Model Loading

The model should be loaded once per process.

Do not reload the model for every query or batch.

Expected lifecycle:

```text
process starts
→ model loads
→ model remains available
→ process exits
```

For the bot process, the model remains loaded.

For a oneshot ingestion worker, memory is released when the process exits.

---

## 13. Lazy Loading

The bot may use lazy loading.

Example:

```text
bot starts
→ exact and fuzzy search available
→ first !qs request
→ model loads
→ subsequent !qs requests reuse model
```

Advantages:

* faster bot startup
* memory is only used when semantic search is needed

Disadvantages:

* first semantic query has higher latency
* first load may time out if not handled carefully

Recommended initial choice:

```text
eager load during bot startup
```

This makes failures visible immediately and avoids surprising the first user with model-download theatre.

---

## 14. Model Cache

Model files must be stored in a persistent directory.

Suggested path:

```text
/var/lib/quote-finder/models
```

Requirements:

* writable by the quote-finder service user
* not stored in `/tmp`
* retained across restarts
* shared between bot and ingestion worker
* protected from accidental deletion
* not committed to source control

Example environment variable:

```text
EMBEDDING_CACHE_DIR=/var/lib/quote-finder/models
```

---

## 15. Model Download

Model download should occur:

* during deployment
* during a dedicated setup command
* or during the first controlled startup

Recommended deployment command:

```bash
quote-finder embeddings download
```

The production bot should not depend on downloading a model during an ordinary user query.

A failed model download must not leave a partial model treated as valid.

---

## 16. Model Revision Pinning

Where supported, the model revision should be pinned.

Store:

```text
model_name
model_revision
```

Example:

```text
model_name = BAAI/bge-small-en-v1.5
model_revision = <specific revision>
```

If the runtime does not expose revision pinning reliably, store:

* package version
* model file checksums
* pipeline version

The goal is reproducibility.

## 17. Document Embedding Input

Document embeddings are generated from semantic chunks.

Each input is:

```text
chunk.text
```

The input should contain:

* original readable story text
* complete paragraph boundaries where possible
* no HTML
* no Discord formatting
* no database metadata
* no chapter URL
* no fic ID

Optional chapter-title prefixes may be evaluated later.

Initial recommendation:

```text
embed chunk body only
```

This avoids artificial title repetition dominating retrieval.

---

## 18. Query Embedding Input

Query embedding uses the semantic-search query.

Example:

```text
Harry stays calm until everyone leaves and then breaks down
```

Before embedding:

* trim whitespace
* collapse repeated whitespace
* remove control characters
* enforce maximum length

Do not apply aggressive quote normalization intended for exact search.

Semantic queries should preserve natural language.

---

## 19. Query Prefixes

Some embedding models benefit from query-specific prefixes or instructions.

The provider interface should allow this behaviour internally.

Example conceptual behaviour:

```text
document:
<story chunk>

query:
Represent this sentence for searching relevant passages:
<user query>
```

The search and ingestion services should not hard-code such prefixes.

The provider is responsible for model-specific formatting.

---

## 20. Document and Query Modes

The provider must use the correct embedding method for each input type.

```text
document chunks
→ embed_documents

scene query
→ embed_query
```

The implementation must not assume both methods are interchangeable.

Model-specific document and query formatting must remain encapsulated in the provider.

---

## 21. Input Size Limits

The embedding provider has a finite token limit.

Chunks must be created small enough to fit.

Recommended chunk configuration remains:

```text
target: 400 words
minimum: 200 words
maximum: 650 words
overlap: 75 words
```

However, word count does not map exactly to token count.

Before embedding, the provider or pipeline should verify that the input is within the supported model limit.

Possible strategies:

* conservative word-based chunking
* tokenizer-based validation
* final truncation safeguard

Silent uncontrolled truncation should be avoided.

---

## 22. Input Truncation

If a chunk exceeds the model’s supported input length:

1. log the chunk ID
2. apply deterministic truncation or rechunking
3. store that truncation occurred
4. avoid crashing the full ingestion if recovery is possible

Preferred behaviour:

```text
rechunk before embedding
```

Fallback:

```text
truncate deterministically
```

Truncation behaviour must be included in the pipeline version.

---

```text
EMBEDDING_DOCUMENT_BATCH_SIZE=12
```

## 24. Query Batch Size

Semantic queries are embedded individually.


## 25. Thread Count

The Netcup VPS has two CPU cores.

Recommended model thread count:

```text
1 or 2
```

Initial default:

```text
EMBEDDING_THREADS=1
```

This leaves capacity for:

* Discord gateway processing
* Postgres queries
* Qdrant requests
* operating-system tasks

For ingestion-only processes, two threads may be tested.

## 26. Ingestion CPU Behaviour

During full ingestion:

* refresh concurrency must remain 1
* document batches must be bounded
* the worker should not use unlimited threads
* the Discord bot must remain responsive
* ingestion may run at lower CPU priority

Possible systemd configuration:

```ini
Nice=10
CPUWeight=25
```

This is optional but recommended if ingestion affects bot latency.

---

## 27. Memory Limits

The embedding pipeline must avoid holding all vectors in memory.

Bad flow:

```text
generate 8,000 vectors
→ keep every vector in RAM
→ upload after completion
```

Preferred flow:

```text
read chunk batch
→ generate vectors
→ validate vectors
→ upload batch to Qdrant
→ release batch memory
→ continue
```

Only a bounded batch should remain in memory.

---

## 28. Streaming Batch Pipeline

Recommended ingestion flow:

```python
for chunk_batch in chunk_repository.iter_batches(
    version_id=version_id,
    batch_size=batch_size,
):
    texts = [chunk.text for chunk in chunk_batch]

    vectors = await embedding_provider.embed_documents(texts)

    points = build_qdrant_points(
        chunks=chunk_batch,
        vectors=vectors,
    )

    await vector_store.upsert_chunks(points)
```

This keeps memory usage predictable.

---

## 29. Vector Validation

Every produced vector must be validated.

Checks:

* vector exists
* vector is a list or supported array
* vector length matches expected dimensions
* all values are finite
* no `NaN`
* no positive infinity
* no negative infinity

Example:

```python
import math


def validate_vector(
    vector: list[float],
    expected_dimensions: int,
) -> None:
    if len(vector) != expected_dimensions:
        raise ValueError("Embedding dimension mismatch")

    if not all(math.isfinite(value) for value in vector):
        raise ValueError("Embedding contains non-finite values")
```

---

## 30. Output Count Validation

For a document batch:

```text
number of input texts
=
number of output vectors
```

A mismatch must fail the batch.

Do not silently discard chunks or pair vectors with the wrong records.

---

## 31. Stable Chunk-to-Vector Mapping

Vectors must remain associated with the correct chunk.

The batch order returned by the provider must be preserved.

Each vector is uploaded with:

```text
chunk_id
fic_id
version_id
chapter_id
chapter_number
chunk_number
paragraph range
embedding model
```

Never rely on text equality alone to identify the chunk.

## 32. Qdrant Vector Configuration

The Qdrant collection must use the same vector dimensions and distance metric as the embedding model.

Initial configuration:

```text
collection:
quote_finder_chunks

vector size:
384

distance:
cosine
```

The exact collection setup is handled by the vector-store implementation.

---

## 33. Collection Compatibility

Before indexing or querying, validate:

```text
Qdrant collection vector size
=
embedding provider dimensions
```

A mismatch must fail immediately.

Do not upload 384-dimensional vectors into a collection configured for another size and wait for Qdrant to express disappointment on your behalf.

---

## 34. One Collection or Multiple Collections

Initial strategy:

```text
one shared Qdrant collection
```

All fics and versions share the collection.

Points are filtered by:

```text
fic_id
version_id
```

A model migration may require:

* a new collection
* a named vector
* or a full collection rebuild

The initial implementation should prefer a new collection for incompatible dimensions.

## 35. Named Vectors

The first release may use one unnamed/default vector.

Future use cases:

* multiple embedding models
* title vectors
* character vectors
* experimental indexes

These are non-goals for the initial implementation.

Suggested fields:

```text
expected_chunk_count

```

This supports:

* progress reporting

## 39. Progress Updates

The ingestion worker should update progress after each successful batch or every few batches.

Example:

```text
embedded_chunk_count = 1200
expected_chunk_count = 5200
```

Avoid one database update per chunk.

Suggested update interval:

```text
every 5 to 10 batches
```

---

## 40. Idempotent Qdrant Upserts

Qdrant point IDs should be deterministic or stable per chunk.

Recommended:

```text
point_id = chunk.id
```

Repeated upserts for the same chunk and version must replace the same point rather than create duplicates.

---

## 41. Embedding Retries

Transient embedding failures may be retried.

Suggested per-batch retry policy:

```text
attempt 1
→ wait 2 seconds

attempt 2
→ wait 5 seconds

attempt 3
→ wait 15 seconds
```

Maximum:

```text
3 attempts
```

Local inference failures are less likely to be transient than network failures.

Retries should not hide:

* out-of-memory errors
* model corruption
* dimension mismatch
* invalid input
* runtime incompatibility

---

## 42. Batch Failure Behaviour

If one batch fails:

1. retry if eligible
2. record batch failure
3. fail the version if unrecoverable
4. do not activate the version
5. retain the previous active version
6. clean incomplete vectors later

The initial implementation should fail the full embedding stage rather than skip chunks.

---

## 43. Partial Resume

A future implementation may resume document embedding.

Possible mechanism:

```text
select chunks where embedding_status != complete
```

Requirements:

* deterministic point IDs
* stable chunk text
* stable embedding model
* idempotent Qdrant upserts

Partial resume is optional for the first release.

---

## 44. Query Embedding Failure

If query embedding fails:

* `!qs` returns an error
* `!qe` remains available
* `!qf` remains available
* the bot remains connected
* the process must not crash

Suggested user-facing state:

```text
Scene search is temporarily unavailable.
```

Internal logs should contain the failure details.

---

## 45. Model Health Check

The embedding provider should expose a health check.

Suggested procedure:

1. embed a fixed short query
2. validate vector dimensions
3. validate finite values
4. return success

Example input:

```text
health check
```

The exact text is irrelevant.

The health check should not run excessively.

---

## 46. Startup Health Check

On bot startup:

1. load the model
2. generate one test query vector
3. validate dimensions
4. verify Qdrant collection compatibility
5. log readiness

If the embedding system fails:

* bot may still start
* exact and fuzzy search remain available
* semantic search is marked unavailable

Recommended behaviour:

```text
degraded startup
```

rather than total bot failure.

---

## 47. Ingestion Health Check

Before beginning full document embedding:

1. verify model loads
2. embed one sample chunk
3. validate dimensions
4. verify Qdrant connectivity
5. verify collection compatibility

This prevents spending minutes parsing a fic before discovering the configured vector size is wrong.

```text

```

## 56. Model Migration

Changing the embedding model requires a new indexed version.

Migration flow:

```text
select new model
→ create new fic version
→ reuse canonical story text
→ rebuild semantic chunks if necessary
→ generate all new vectors
→ upload to compatible Qdrant collection
→ validate
→ atomically activate
→ retain old version temporarily
```

Do not modify vectors belonging to the active version in place.

---

## 57. Dimension Changes

If the new model uses different dimensions:

```text
384 → another size
```

the migration must use:

* a new Qdrant collection
* or another explicitly compatible vector configuration

Do not mix incompatible vector sizes in one unnamed vector collection.

---

## 58. Same-Dimension Model Changes

Even if two models both produce 384-dimensional vectors, they are not automatically compatible.

Model identity must still match.

Vectors from different models occupy different semantic spaces.

A dimension match is necessary but not sufficient.

---

## 59. Rollback

Rollback to an older fic version requires:

* old Neon version data
* old Qdrant vectors
* old embedding metadata
* query provider capable of loading the old model

If the old model is no longer available locally, semantic search for the rolled-back version may not work.

Therefore, retained rollback versions should retain model availability where practical.

## 62. Model Comparison

Before indexing production permanently, compare at least:

* the chosen BGE small baseline
* one other small FastEmbed-compatible English model
* optionally one stronger but larger model

Use the same:

* chunks
* benchmark queries
* Qdrant distance
* retrieval count

Do not choose solely from a public leaderboard.

The correct model is the one that finds the user’s remembered fanfiction scenes.

Embedding quality must be tested together with chunking.

Variables include:

* chunk target size
* overlap
* paragraph boundaries
* chapter-title inclusion
* dialogue-heavy passages
* scene transitions

A better model cannot fully compensate for chunks that splice two unrelated scenes together with the grace of a malfunctioning stapler.

## 69. Out-of-Memory Handling

If the process runs out of memory:

1. fail the current embedding job
2. do not activate the fic version
3. preserve the previous active version
4. reduce batch size
5. restart the worker if necessary
6. record failure details

The worker should not continue after severe memory corruption or process termination.

---

## 70. Graceful Degradation

If the local model cannot load:

```text
!qe
→ remains available

!qf
→ remains available

!qs
→ unavailable
```

The bot should report degraded readiness.

Semantic failure must not take down quote search.

---

## 71. Security

The embedding system must:

* use trusted model sources
* pin model identity where possible
* avoid loading arbitrary user-supplied models
* store model files outside the repository
* restrict model-cache permissions
* validate model output
* avoid logging full story text
* prevent unbounded query length
* avoid exposing local filesystem paths in Discord errors

## 78. Document Embedding Acceptance Criteria

Document embedding is complete when:

1. the local model loads from persistent cache
2. semantic chunks can be embedded in batches
3. batch size is configurable
4. thread count is bounded
5. vector count matches chunk count
6. vectors have exactly 384 dimensions
7. vectors contain only finite values
8. vectors are uploaded incrementally to Qdrant
9. chunk-to-vector mapping is correct
10. progress is recorded
11. failed embedding prevents activation
12. partial vectors can be cleaned safely
13. the active fic version remains unchanged on failure

---

## 79. Query Embedding Acceptance Criteria

Query embedding is complete when:

1. `!qs` queries are embedded locally
2. the Discord event loop is not blocked directly
3. query vectors use the same model as document vectors
4. vector dimensions are validated
5. query timeout is enforced
6. concurrent query embedding is bounded
7. model failures affect only semantic search
8. exact and fuzzy search remain available
9. query embedding latency is logged
10. the query vector can be used against the active Qdrant index

---

## 80. Compatibility Acceptance Criteria

Embedding compatibility is complete when:

1. every fic version stores embedding metadata
2. semantic search checks model identity
3. semantic search checks dimensions
4. Qdrant collection size is validated
5. incompatible models are rejected
6. model changes trigger re-embedding
7. same-dimension models are not assumed compatible
8. old and new vectors are never mixed in one active version
9. version activation requires complete vectors
10. rollback preserves embedding compatibility

---

## 81. Operational Acceptance Criteria

The local embedding deployment is complete when:

1. model files survive service restarts
2. the model can be downloaded during deployment
3. the bot starts in degraded mode if embeddings fail
4. health checks expose model readiness
5. document ingestion does not block Discord commands
6. batch ingestion stays within VPS memory limits
7. query embedding is responsive under normal load
8. full refresh concurrency remains bounded
9. logs identify embedding failures clearly
10. the provider can later be replaced without rewriting search commands

## 82. Final Embedding Rules

```text
Story chunks
→ embedded locally
→ uploaded to Qdrant

!qs query
→ embedded locally
→ compared against the active fic version

Document and query vectors
→ same provider
→ same model
→ same revision
→ same dimensions

Embedding failure
→ semantic search unavailable
→ exact and fuzzy search remain available

Model change
→ create replacement version
→ re-embed
→ validate
→ atomically activate
```

The initial provider is local FastEmbed-compatible ONNX inference.

The initial model is `BAAI/bge-small-en-v1.5`.

The model runs on the Netcup VPS.

Qdrant stores vectors.

Neon stores canonical text and embedding metadata.

The embedding provider remains replaceable.

The Discord bot embeds queries only.

Large document embedding belongs in the ingestion worker.
